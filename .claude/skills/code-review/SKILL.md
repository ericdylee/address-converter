# 코드 리뷰 스킬

이 스킬은 코드 변경사항을 **체계적으로** 리뷰합니다. 인상 비평이 아니라 항목별 체크리스트로 점검하고, 발견한 이슈를 **심각도(severity)** 별로 정리해서 보고합니다.

## 리뷰 대상 정하기

먼저 무엇을 리뷰할지 명확히 합니다. 우선순위 순:

1. **명시적 범위가 주어진 경우** — 사용자가 지정한 파일/디렉토리/PR
2. **스테이징된 변경** — `git diff --cached`
3. **워킹 디렉토리 변경** — `git diff`
4. **최근 커밋** — `git log -1 --stat` + `git show HEAD`
5. **위 모두 비어있으면** — 사용자에게 어느 범위를 리뷰할지 질문

> ⚠️ **전체 코드베이스를 통째로 리뷰하지 말 것.** 범위가 모호하면 사용자에게 묻습니다.

## 점검 카테고리 (체크리스트)

각 항목을 순서대로 확인합니다. 해당 사항 없으면 "해당 없음"으로 명시.

### 1. 정확성 (Correctness)
- [ ] 의도한 동작과 코드가 일치하는가?
- [ ] 엣지 케이스 처리 (빈 입력, null/undefined, 0, 음수, 매우 큰 값)
- [ ] 비동기 흐름의 race condition / 누락된 await
- [ ] off-by-one, 잘못된 조건문, 부정확한 비교 연산자

### 2. 보안 (Security)
- [ ] 사용자 입력이 검증/이스케이프 없이 사용되는가? (XSS, SQL injection, command injection)
- [ ] 비밀값(API 키, 토큰, 비밀번호)이 코드/로그/응답에 노출되는가?
- [ ] CORS, CSRF, 인증/인가 누락
- [ ] 외부 API 응답을 신뢰해서 그대로 렌더링하거나 실행

### 3. 에러 처리
- [ ] try/catch 없이 외부 호출 (fetch, DB, 파일 IO)
- [ ] 에러를 그냥 삼키지(swallow) 않고 의미 있는 메시지로 변환했는가?
- [ ] 사용자에게 노출되는 에러 메시지가 내부 스택 트레이스를 누설하지 않는가?
- [ ] Promise rejection이 처리되지 않는 경로

### 4. 타입 안정성
- [ ] `any`, `as`, `// @ts-ignore` 남용
- [ ] 외부 데이터(API 응답, JSON.parse)를 타입 단언 없이 신뢰하는 코드
- [ ] 선언된 타입과 실제 값의 불일치

### 5. 가독성 & 유지보수성
- [ ] 함수/파일이 한 가지 책임만 갖는가? (긴 함수 = 분리 신호)
- [ ] 마법의 숫자/문자열을 상수로 빼야 하는가?
- [ ] 변수/함수 이름이 의도를 드러내는가? (`data`, `temp`, `handle` 등 막연한 이름)
- [ ] 중복 코드 (3번째 등장하면 추출 검토)

### 6. 성능
- [ ] N+1 쿼리, 불필요한 반복문, 무한 리렌더링
- [ ] 큰 데이터를 메모리에 다 적재 vs 스트리밍/페이지네이션
- [ ] 디바운싱/스로틀링 누락 (사용자 입력, 스크롤 이벤트)

### 7. 컨벤션 & 일관성
- [ ] 프로젝트 기존 스타일/패턴과 어긋나는가? (먼저 `CLAUDE.md`, 인접 파일 확인)
- [ ] 명명 규칙 (camelCase / PascalCase / kebab-case)
- [ ] 임포트 순서, 절대/상대 경로 혼용

### 8. 테스트 (해당 시)
- [ ] 새 로직에 테스트가 있는가?
- [ ] 테스트가 의미 있는 검증을 하는가, 아니면 단순 호출 확인뿐인가?
- [ ] 엣지 케이스 테스트 누락

### 9. 문서 (해당 시)
- [ ] 공개 API(라이브러리 함수, 라우트)에 사용법 설명이 필요한가?
- [ ] README/CLAUDE.md가 변경된 동작과 맞는가?

## 심각도 (Severity)

발견 항목은 반드시 다음 4단계 중 하나로 분류:

| 등급 | 의미 | 예시 |
|------|------|------|
| 🔴 **Critical** | 머지 전에 반드시 수정. 버그/보안/데이터 손실 | API 키 노출, SQL injection, 무한 루프 |
| 🟡 **High** | 곧 수정해야 함. 명백한 결함이지만 즉각 장애는 아님 | 처리되지 않는 에러, 잘못된 타입, 누락된 await |
| 🔵 **Medium** | 개선 권장 | 가독성, 컨벤션 불일치, 작은 중복 |
| ⚪ **Low / Nit** | 취향에 가까움. 무시해도 무방 | 변수명 제안, 줄바꿈, 코멘트 |

## 출력 형식

리뷰 결과는 반드시 아래 구조로 출력:

```markdown
## 리뷰 요약
- **범위**: <어떤 파일/diff를 봤는지>
- **총 발견**: 🔴 N건 / 🟡 N건 / 🔵 N건 / ⚪ N건
- **머지 가능 여부**: ✅ Ready / ⚠️ 조건부 / ❌ 차단

## 발견 항목

### 🔴 Critical
1. **[file:line] 제목**
   - 문제: ...
   - 영향: ...
   - 제안: ```diff 또는 코드 예시 ```

### 🟡 High
...

### 🔵 Medium
...

### ⚪ Nits
- file:line — 한 줄 설명 (Nits는 압축 표시)

## 잘된 점 (Positive)
- ...

## 다음 단계
- [ ] Critical/High 우선 수정
- [ ] (선택) Medium 반영
```

## 리뷰 시 지켜야 할 원칙

1. **인상이 아니라 근거.** "이게 더 좋아 보임" ❌ → "X 케이스에서 Y 동작이 어긋남" ⭕
2. **위치를 정확히.** 항상 `file:line` 형태로 인용.
3. **수정 제안은 구체적으로.** 가능하면 `diff` 또는 대체 코드 한 토막.
4. **칭찬도 적어둘 것.** 잘 작성된 부분은 명시 — 다음에도 그렇게 쓰라는 신호.
5. **사용자를 평가하지 말 것.** 코드를 평가합니다. "왜 이렇게 짰냐"가 아니라 "이 부분은 ~~ 이유로 ~~로 바꾸는 게 안전".
6. **범위 밖은 건드리지 말 것.** 리뷰 범위 밖의 코드를 리팩토링 제안하지 않음.

## 안티 패턴

다음은 하지 말아야 할 리뷰:

- ❌ "전반적으로 잘 짰습니다" (구체성 없음)
- ❌ 모든 변수명을 trivial하게 지적 (노이즈)
- ❌ 잘못된 정보를 자신 있게 단언 (확실하지 않으면 "확인 필요"로 표기)
- ❌ 한 번에 30+ 항목을 쏟아내기 (우선순위 없음 = 무시됨). 상위 10개로 압축, 나머지는 Nits로.
